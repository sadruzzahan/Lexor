# Lexor v2 — Merger Plan & Architecture

> **Purpose:** Unify three sibling repos — `Beat` (detective field kit), `Briefcased` (criminal-defense copilot), and `Lexor` (consumer legal-help) — into a single product called **Lexor**, an end-to-end justice/legal-tech platform spanning citizen intake → investigation → defense.
>
> **Status:** Plan only. Execution gated on owner approval. None of the three source repos should be deleted or rewritten until the merged product passes the parity checklist in §10.
>
> **Author:** Drafted 2026-05-04 after deep code audit of all three repos.

---

## 1. Why merge — the evidence

Reading the actual schemas reveals the merge is mostly **already done in spirit**:

| Concern | Beat | Briefcased | Lexor | Compatibility |
|---|---|---|---|---|
| `users` table shape | id (text UUID), displayName, email, tier | id (uuid), displayName, email, googleRefreshToken | implicit Clerk userId (text) | **High** — same fields, ID type just needs unification |
| `cases.rolePack` | `text default "detective"` | `text check ('defender','detective')` | not present (uses `vertical`) | **Briefcased was explicitly designed for both packs** |
| `cases.jurisdictionContext` | `jsonb` | `jsonb` | string `jurisdiction` | Easy bridge |
| `cases.status` | enum {open, closed, archived} | text default "created" | enum {queued, parsing, analyzing, drafting, complete, failed} | **Conflict — must resolve (see §3.2)** |
| Auth provider | Clerk | Google OAuth | Clerk | **Standardize on Clerk** |
| AI providers | Anthropic + Gemini + OpenAI via Replit AI Integrations | Vercel AI SDK over Replit AI Integrations | Anthropic via custom client | **Standardize on Vercel AI SDK over Replit AI Integrations** |
| API spec | OpenAPI 3.1 + Orval | mostly hand-written Express | OpenAPI 3.1 + Orval | **Standardize on OpenAPI 3.1 + Orval** |
| Frontend router | Wouter | Wouter | react-router | **Standardize on Wouter** |

**Conclusion:** Beat + Briefcased are 80% schema-aligned — the merge is largely mechanical. Lexor adds a third role (`complainant` / consumer) that requires bridging a different case lifecycle but reuses the same user, file, run, and artifact infrastructure.

---

## 2. Product vision (one paragraph for the README)

> **Lexor** is an end-to-end justice platform. A citizen reports a problem (eviction, debt collection, wage theft, unsafe landlord) and Lexor's `complainant` track grounds the facts against statute, drafts response letters, and files regulator complaints. If the case escalates, an investigator picks it up in the `detective` track — multi-agent workflows gather scene evidence, witness statements, and chain-of-custody-clean artifacts. When it reaches a courtroom, the lawyer's `defender` track takes over — Brady review, jury selection notes, plea-path analysis, and courtroom rehearsal. One database, one auth model, one design system, three role packs.

### 3.2 Unified database schema

The most critical merge work. Source-of-truth schema lives in `lib/db/src/schema/`.

**Identity & sessions**
```ts
// users.ts — uuid IDs (Briefcased style), Clerk-provisioned
export const users = pgTable("users", {
  id: uuid("id").primaryKey().defaultRandom(),
  clerkUserId: text("clerk_user_id").unique().notNull(),
  displayName: text("display_name"),
  email: text("email").unique(),
  primaryRolePack: rolePackEnum("primary_role_pack").notNull().default("complainant"),
  tier: tierEnum("tier").notNull().default("free"),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
});

export const rolePackEnum = pgEnum("role_pack", ["complainant", "detective", "defender"]);
export const tierEnum = pgEnum("tier", ["free", "pro", "agency"]);
```

> `clerkUserId` replaces the legacy text userIds in Beat/Lexor; one-time backfill script writes existing rows.

**Unified cases**
```ts
// cases.ts — unifies all three case shapes
export const cases = pgTable("cases", {
  id: uuid("id").primaryKey().defaultRandom(),
  userId: uuid("user_id").notNull().references(() => users.id),

  rolePack: rolePackEnum("role_pack").notNull(),
  vertical: caseVerticalEnum("vertical"),                     // Lexor's enum, only required for complainant
  title: text("title").notNull(),
  description: text("description"),
  jurisdictionContext: jsonb("jurisdiction_context"),         // jsonb everywhere, Lexor's flat string is migrated in
  language: text("language").notNull().default("en"),

  // Two-axis status: lifecycle + workflow
  lifecycleStatus: caseLifecycleEnum("lifecycle_status").notNull().default("draft"),
  workflowState: text("workflow_state"),                      // free-form per rolePack (e.g. Lexor's "parsing" / "analyzing")

  // Track-specific fat columns kept opt-in to avoid wide-row pollution
  complainantParsed: jsonb("complainant_parsed"),
  complainantViolations: jsonb("complainant_violations"),
  complainantResponseLetter: jsonb("complainant_response_letter"),
  complainantRegulatorComplaints: jsonb("complainant_regulator_complaints"),
  embedding: vector("embedding", { dimensions: 1024 }),

  adversaryEntityId: uuid("adversary_entity_id").references(() => entities.id, { onDelete: "set null" }),

  deletedAt: timestamp("deleted_at", { withTimezone: true }),
  createdAt: timestamp("created_at", { withTimezone: true }).defaultNow().notNull(),
  updatedAt: timestamp("updated_at", { withTimezone: true }).defaultNow().notNull(),
});

export const caseLifecycleEnum = pgEnum("case_lifecycle", [
  "draft", "active", "in_review", "closed", "archived", "failed",
]);

export const caseVerticalEnum = pgEnum("case_vertical", [
  "eviction", "debt", "wage", "contract", "criminal", "other",
]);
```

> `lifecycleStatus` collapses the three legacy status enums into one shared lifecycle. `workflowState` is free-text-per-rolePack so each track can keep its sub-states (Lexor's parsing/analyzing, Briefcased's brady-review, etc.) without contaminating the shared enum.

**Tables to copy verbatim with minimal renames**

| New | From | Notes |
|---|---|---|
| `case_files` | Beat `case-files` | Add `rolePack` index for fast filtering |
| `runs` | Beat `runs` | Multi-track |
| `run_events` | Beat `run-events` | Used for SSE replay |
| `artifacts` | Beat `artifacts` | Generic per-run artifacts |
| `chain_of_custody` | Beat `chain-of-custody` | Detective track only, but shared schema |

| `coalitions` + `coalition_members` + `coalition_votes` | Lexor | Complainant track |

**Migrations:** start fresh. `lib/db/migrations/0000_lexor_v2_init.sql` is a new baseline. Backfill scripts (one per source repo) replay existing data into the unified schema; details in §6.

### 3.3 Unified auth — Clerk

- **Drop Briefcased's Google-OAuth-only flow.** Replace with Clerk's Google social provider (config-only, zero code in our app).
- Encrypted `googleRefreshToken` columns are removed; Clerk owns the refresh cycle.
- `auth.ts` middleware is a single Clerk Express middleware that resolves `req.auth.userId` → `users.clerkUserId` → DB `users.id`. Provision-on-first-request (Beat's pattern) is preserved.
- Demo mode: keep Briefcased's `x-demo-user` header bypass, gated on `NODE_ENV !== "production"`.

### 3.4 Unified AI provider stack

- **Vercel AI SDK** as the single client (Briefcased's pattern wins — broadest ecosystem).
- All providers (Anthropic / OpenAI / Gemini) routed through **Replit AI Integrations** (`AI_INTEGRATIONS_*` env vars). No raw API keys.
- Lexor's bespoke Anthropic client and Beat's mixed-direct providers both retire.
- Beat's **Mastra** workflow engine becomes a separate `lib/agent-runtime` package — used by all three tracks for any multi-step agent flow (Brady review, regulator filing, witness mapping all become Mastra workflows).

### 3.5 Unified API surface

`lib/api-spec/openapi.yaml` becomes the single contract. Routes reorganize as:

```

# Track-specific
```

### 3.6 Frontend — single SPA, role-based shell

`artifacts/lexor-web/src/App.tsx`:

```tsx
<Switch>
  <Route path="/" component={Landing} />
  <Route path="/auth/*" component={ClerkAuth} />
  <Route path="/onboarding" component={RolePackPicker} />

  <Route path="/c/*" component={ComplainantShell} />   {/* /c/cases, /c/regulators, /c/coalitions */}
  <Route path="/d/*" component={DetectiveShell} />     {/* /d/cases, /d/runs, /d/witnesses */}

  <Route path="/share/:token" component={ShareView} />

```

A user's `primaryRolePack` decides their default landing page, but they can switch tracks per-case. The case detail page renders different panels based on `case.rolePack`.

**Design system:** consolidate to one shadcn-based component library in `artifacts/lexor-web/src/components/ui/`. Beat's halo-animated `AgentPane` and Briefcased's Framer Motion patterns become shared primitives.

## 4. Migration phases

Each phase ends in a working, deployable build. Do not start phase N+1 until N's parity check passes.

### Phase 0 — Bootstrap (1–2 days)
- Create new `lexor` repo (or `lexor-v2` branch in current Lexor repo).
- Copy the pnpm-workspace template; create the empty `lib/db`, `lib/api-spec`, `lib/api-client-react`, `lib/api-zod`, `artifacts/api-server`, `artifacts/lexor-web`, `artifacts/mockup-sandbox`.
- Wire Clerk (auth), Drizzle (DB), Replit AI Integrations (providers).
- Define `lib/db/src/schema/index.ts` with the §3.2 schema. `pnpm --filter @workspace/db run push` against a fresh Postgres.
- **Parity check:** `pnpm run typecheck && pnpm run build` succeed; healthz returns 200.

### Phase 1 — Defender track first (was Briefcased) (4–7 days)
- Defender is the most surface area (147 pages) and the proven-flagship candidate, so go first to validate the architecture.
- Migrate Briefcased's routes into `artifacts/api-server/src/routes/defender/`, replacing the inline auth with Clerk, the Google-OAuth code with Clerk's Google provider, and the bespoke AI calls with the unified Vercel AI SDK provider stack.
- Migrate Briefcased's React app into `artifacts/lexor-web/src/routes/(defender)/`. Convert react-router to Wouter.
- Backfill data: `scripts/backfill-from-briefcased.mjs` reads Briefcased's `users` and `cases` tables, normalizes IDs, writes into the new schema.
- **Parity check:** every Briefcased UI flow still works. OCR, PDF export, courtroom rehearsal, agent console, Brady review — all green.

### Phase 2 — Detective track (was Beat) (3–5 days)
- Migrate Beat's routes into `artifacts/api-server/src/routes/detective/`. Mastra orchestrator moves to `lib/agent-runtime`.
- Migrate Beat's frontend into `artifacts/lexor-web/src/routes/(detective)/`. The 4-pane agent dashboard becomes a generic component used by any track that needs SSE-streamed multi-agent runs (Lexor's complainant pipeline will reuse it in Phase 3).
- Backfill Beat data via `scripts/backfill-from-beat.mjs`.
- **Parity check:** Beat's full flow (case create → run → witness/scene/suspect agents → draft → share) still works end-to-end.

### Phase 3 — Complainant track (was Lexor) (5–8 days)
- Largest schema additions: `entities`, `coalitions`, `coalition_members`, `coalition_votes`, `disclosures`, `lawyer_bids`, `map_markers`, `trials`. Add migrations.
- Port Lexor's SSE pipeline (vision → classify → grounding → rules → draft → complaints → embedding → adversary → coalition) into a Mastra workflow under `lib/agent-runtime/workflows/complainant-pipeline.ts`.
- Port the curated CA/TX/NY/federal statute corpus into `lib/legal-corpus`.
- Port Twilio voice + WhatsApp webhooks into `artifacts/api-server/src/routes/complainant/voice.ts` and `whatsapp.ts`.
- Port pgvector embedding logic into `cases.embedding` column (already in schema).
- Migrate Lexor's frontend into `artifacts/lexor-web/src/routes/(complainant)/`.
- Backfill via `scripts/backfill-from-lexor.mjs`.
- **Parity check:** consumer intake → regulator filing flow works; coalition voting works; Twilio webhook smoke tests green.

### Phase 4 — Cross-track features (3–5 days)
- **Case handoff:** a complainant case can be promoted to a detective case (preserves files, citations, jurisdiction) and from there to a defender case. UI: "Refer to investigator" / "Refer to lawyer" buttons.
- **Lawyer bid marketplace:** complainant cases can solicit defender bids (uses `lawyer_bids` table from Lexor).
- **Unified inbox/notifications:** consolidate Lexor's `inbox` + `notifications` into a single `notifications` stream visible across tracks.
- **Shared search:** pgvector-backed similarity search across all cases the user has access to.

### Phase 5 — Polish & deploy (2–4 days)
- E2E tests via Playwright (Beat already has the harness; reuse).
- Vercel deployment for `lexor-web` (hostname `lexor.app` or similar).
- Render/Fly deployment for `api-server` (or Replit deployment).
- Smoke tests in production.
- **Cutover criterion:** sign in with Clerk, create a case in each rolePack, run the SSE pipeline in each, no errors. Then archive Beat / Briefcased / old Lexor with READMEs pointing to the new repo.

**Total estimated effort: 18–31 days of focused work.** Realistically with interruptions: 4–7 weeks calendar.

## 6. Backfill scripts (sketch)

```ts
// scripts/backfill-from-briefcased.mjs
import { drizzle } from "drizzle-orm/node-postgres";
const oldDb = drizzle(process.env.BRIEFCASED_DATABASE_URL!);
const newDb = drizzle(process.env.LEXOR_DATABASE_URL!);

// 1. Users — uuid → uuid, displayName + email pass through
const oldUsers = await oldDb.execute(`SELECT id, display_name, email, created_at FROM users`);
for (const u of oldUsers) {
  await newDb.insert(usersTable).values({
    id: u.id,                                  // preserve uuid
    clerkUserId: `legacy-briefcased-${u.id}`,  // placeholder until user re-auths
    displayName: u.display_name,
    email: u.email,
    primaryRolePack: "defender",
    createdAt: u.created_at,
  }).onConflictDoNothing();
}

// 2. Cases — preserve id, set rolePack=defender
// ...
```

Each backfill script has a `--dry-run` mode that reports row counts and conflicts without writing.
