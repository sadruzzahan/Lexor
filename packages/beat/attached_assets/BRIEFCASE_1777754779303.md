## 1. Project Proposal

### 1.1 Vision Statement

Briefcase gives every criminal-defense lawyer in the world a multi-agent investigation team in their pocket: it reads the case files they already have, in the language they were written in, and produces a defensible, source-grounded preparation pack — timeline, evidence gaps, cross-examination questions, and precedent — in roughly ninety seconds.

### 1.2 Problem Statement

| Dimension | Reality |
|---|---|
| **Volume** | In the United States, public defenders represent ~80% of criminal defendants. Median time per file: under ten minutes. The same pattern repeats globally: UK duty solicitors, Brazilian defensores públicos, Indian legal-aid lawyers, French commis d'office, Italian patrocinatori d'ufficio, etc. |
| **Tooling gap** | The most-funded legal AI products as of May 2026 — Harvey ($11B raise, March 2026), CoCounsel (Thomson Reuters), Vincent (Clio, April 2026), Paxton — target BigLaw and general practice. None target public defense as a primary audience. |
| **Form-factor gap** | The few products that *do* serve public defense (e.g. JusticeText) are web-only single-purpose transcription tools. None are mobile-first multi-agent investigation systems. |
| **Trust gap** | Existing chat-style legal AI hallucinates citations. 600+ documented sanctions for AI-fabricated authority since 2023 (Mata v. Avianca onward). Lawyers will not adopt tools that risk professional discipline. |
| **Geographic gap** | Even the best products are US-and-UK-centric. There is no global legal AI built to be jurisdiction-agnostic by architecture rather than by patchwork integration. |

### 1.3 Target Users

| Segment | Description |
|---|---|
| **Primary** | Public defenders, court-appointed counsel, criminal-defense solos, and small-firm criminal-defense lawyers worldwide. |
| **Persona — Maria** | 34, public defender, Cook County, IL. Caseload: 110 active cases. Phone is her primary work device between courthouse, jail, and home. ~8 minutes per file on average. |
| **Persona — Aanya** | 29, legal-aid lawyer, Mumbai. Works in Marathi and English. Sessions Court schedule is brutal; she preps on the local train using her phone. |
| **Persona — Jamal** | 41, duty solicitor, Birmingham UK. Police-station representation requires fast disclosure review on a phone in a custody suite. |
| **Persona — Lucas** | 38, defensor público, São Paulo. Documents arrive in Portuguese; needs to extract contradictions across dozens of pages quickly. |

### 1.4 Goals & Non-Goals

**Goals**

1. Reduce average per-file preparation time by an order of magnitude (ten minutes → one minute) without reducing quality.
2. Surface contradictions and exculpatory inconsistencies a human might miss across discovery documents.
3. Produce cross-examination questions and precedent leads that are always grounded in either uploaded documents or verified live sources.
4. Work the same way in any country, in any language, with no user-side configuration.
5. Show the user what the AI is doing, in plain sight, the entire time it is working.

**Non-Goals**

1. Briefcase does not give legal advice. The lawyer makes every legal decision.
2. Briefcase does not operate during live courtroom proceedings or live testimony.
3. Briefcase does not store or replay live audio of opposing counsel, witnesses, or judges.
4. Briefcase does not maintain its own legal database. It grounds in the user's documents and verified live sources.
5. Briefcase does not auto-file motions or auto-submit to courts.
6. Briefcase does not provide ranked predictions of case outcomes.

### 1.5 Value Proposition

> "The same preparation a paralegal would take two hours to do, on the phone, in ninety seconds, in any language, with every claim traceable to a source."

### 1.6 Success Metrics (post-launch, not buildathon-blocking)

| Metric | Target |
|---|---|
| Time-to-first-useful-output | < 90 seconds median |
| Citation verification rate | ≥ 95% of citations shown to user pass the on-stage verifier |
| User-reported errors per case | ≤ 1 in 20 cases triggers user correction |
| Daily active use per lawyer | ≥ 3 cases per active workday |
| Country coverage | App functions for users in ≥ 30 jurisdictions in beta |

### 1.7 Scope of This Document

This document fully specifies Briefcase. It includes shared infrastructure (the JusticeOS backend engine, the agent protocol, the database schema, the deployment design) wherever Briefcase depends on it, so the document is self-contained. The same backend also serves Beat; that overlap is documented identically in `BEAT.md` so neither document requires the other to be implemented end-to-end.

### 1.8 Stakeholders

| Stakeholder | Interest |
|---|---|
| **End users** (defenders) | Tool that saves time without risking sanctions |
| **Bar regulators** | Output quality, citation accuracy, ethics compliance |
| **Court systems** | Whether AI-assisted briefs maintain accuracy standards |
| **Replit Buildathon judges** | Originality, technical depth, impact, fit with Replit's 2026 product story |

2. Users have intermittent internet connectivity sufficient to stream agent responses (≥ 1 Mbps).
3. Users either upload documents directly or have Google Drive / equivalent cloud storage.

5. Replit's AI integrations (Anthropic, Gemini, OpenAI) and Replit Postgres are available for the build.

## 2. Feasibility Study

### 2.1 Technical Feasibility — Verdict: **HIGH**

| Factor | Assessment |
|---|---|
| **Multi-agent orchestration on TypeScript** | Mastra is production-ready as of May 2026 and is the same framework Replit Agent 3 is built on. Vercel AI SDK 5 handles streaming + tool-call protocol cleanly. Risk: low. |
| **Server-Sent Events on React Native** | `react-native-sse` is mature, has automatic reconnection, and works behind every proxy used in dev/prod. Risk: low. |
| **PDF parsing + OCR pipeline** | E2B sandbox can run any Python package on demand (PyMuPDF, pdfplumber, Tesseract, EasyOCR, paddleOCR). Native fallback uses Apple VisionKit (iOS) or ML Kit (Android). Risk: low. |
| **Google Drive ingest** | Google Workspace MCP server (managed by Google as of Cloud Next '26) exposes drive.list / drive.read / drive.search. No bespoke REST wrapper needed. Risk: low–medium (OAuth scope handling on dev builds). |
| **Streaming agent UI on mobile** | Vercel AI Elements + AI SDK 5 give us Conversation, Message, Tool, Reasoning, CodeBlock, Task primitives. Reanimated 4 + Skia handle the visual layer at 120 fps. Risk: low. |
| **Citation verification in sandbox** | E2B Firecracker microVMs cold-start in ~150 ms; we can run a verifier per citation and parallelize. Risk: low. |
| **Multi-language input/output** | Claude Sonnet 4.6 and Gemini 3.x both natively handle 100+ languages. OCR engines support Latin, Cyrillic, Devanagari, Han, Arabic, etc. Risk: low. |

### 2.2 Operational Feasibility — Verdict: **HIGH**

The user workflow (open app → ingest documents → watch agents work → tap a result → save to prep) maps cleanly onto existing mobile interaction patterns the user already knows. No new metaphor is introduced. The "visible agents" pane reads like Replit Agent / Cursor Composer / Claude Code, which are widely recognized in 2026.

### 2.3 Legal & Regulatory Feasibility — Verdict: **HIGH (with disclosure)**

| Jurisdiction concern | Briefcase's posture |
|---|---|
| **ABA Formal Opinion 512 (US)** | Permits AI for research and analysis when the lawyer verifies output. Briefcase enforces verification by exposing every source and verified citation. |
| **EU AI Act high-risk categories** | "Administration of justice" Article 6 list applies to systems used **by judicial authorities or on their behalf**. A defense-side preparation tool is *not* in that scope. We add an explicit disclosure banner regardless. |
| **Bar rules on confidentiality (US Model Rule 1.6, EU equivalents)** | Documents are processed in Replit-hosted infrastructure. We disclose sub-processors (Anthropic, Google, OpenAI, E2B) in an in-app data-handling page. Production will add an enterprise tier with self-host or BAA. |
| **Public-records limits (Brady, CPIA)** | Briefcase only reads what the lawyer uploads or pulls from their own Drive. It never accesses opposing-party systems. |
| **Cross-border data transfer** | Documented per-jurisdiction in the in-app compliance banner. EU users see a GDPR notice. Indian users see DPDP 2023 notice. Brazilian users see LGPD notice. |

### 3.2 Overall Description

#### 3.2.1 Product Perspective
Briefcase is a **client of** the JusticeOS engine. The engine is a server-side TypeScript application using Hono, Mastra, Vercel AI SDK 5, and an OpenAPI-defined contract. Briefcase consumes that contract through generated React Query hooks. The engine in turn integrates with Google Workspace MCP (for Drive), E2B (for sandboxed compute), Replit AI integrations (for LLM access), and Replit Postgres (with pgvector).

#### 3.2.2 Product Functions (high level)
1. Create and manage cases.
2. Ingest case files from device camera (with OCR), local upload, or Google Drive (via MCP).
3. Auto-detect jurisdiction and language from ingested documents.
4. Run a planner agent that decomposes the goal into a defender role pack.
5. Run subagents in parallel: TimelineBuilder, EvidenceGapAuditor, CrossExaminationGenerator, PrecedentFinder.
6. Stream subagent reasoning, tool calls, and partial results to the mobile client over SSE.
7. Render a 4-pane visible agent dashboard with synchronized live updates.
8. Verify every cited authority via an on-stage verifier in E2B before display.
9. Persist case state, run history, and per-subagent artifacts.
10. Display source-grounded results with tap-through to the underlying PDF page anchor.

#### 3.2.3 User Characteristics
| Characteristic | Description |
|---|---|
| Technical skill | Low to moderate. Comfortable with chat-style apps and document scanners. Not assumed comfortable with shells, APIs, or developer tools. |
| Devices | Modern smartphone (iOS 17+ / Android 12+). May or may not own a tablet/laptop. |
| Languages | Any language the LLMs support (100+). UI ships English; localized UI is post-launch. |
| Workplace | Often offline-tolerant environments (courthouses, jails, transit) — graceful degradation required. |

#### 3.2.4 Constraints
| Constraint | Source |
|---|---|
| iOS minimum: 17.0 | Skia + New Architecture compatibility |
| Android minimum: API 31 (Android 12) | New Architecture, ML Kit text recognition |
| Backend runtime: Node 20 LTS | pnpm-workspace template baseline |
| OpenAPI contract is single source of truth | All client/server types are generated; no hand-written DTOs |
| No live-audio capture during legal proceedings | Bar ethics + jurisdiction recording laws |
| All citations shown to user must pass the verifier | Anti-hallucination guarantee |

#### 3.2.5 Assumptions and Dependencies
- Replit Postgres availability with `pgvector` extension installed.
- Replit AI integrations (Anthropic, Gemini, OpenAI) configured.
- Secrets present: `E2B_API_KEY`, `TAVILY_API_KEY`, `GOOGLE_OAUTH_CLIENT_ID_IOS`, `GOOGLE_OAUTH_CLIENT_ID_ANDROID`, `GOOGLE_OAUTH_CLIENT_ID_WEB`, `GOOGLE_OAUTH_CLIENT_SECRET`, `SESSION_SECRET`.
- EAS Build account for iOS/Android dev clients (Expo Go cannot run native modules required: doc scanner, Google sign-in, Skia New Arch).

### 3.3 Functional Requirements

Format: **FR-XXX** — Brief title — Description — Acceptance criteria.

#### Onboarding & Auth
- **FR-001 — Demo sign-in.** App presents a single "Continue as Demo Lawyer" button on first launch. Tapping it sets `userId = demo_user_pd` via the `x-demo-user` header on subsequent requests. *Acceptance:* user reaches Cases home in ≤ 2 taps from cold start.
- **FR-002 — Google sign-in (post-demo).** App offers Google sign-in for production users. Stores refresh token server-side, encrypted at rest. *Acceptance:* user can authorize Drive scope, return to app, and see their Drive folders. (Out of demo path; in scope for Briefcase release build.)

#### Case Lifecycle
- **FR-010 — Create case.** User can create a new case from the Cases home via FAB (floating action button). User must choose one of three ingest methods on the New Case sheet: Scan documents, Pick from Drive, Type a goal. *Acceptance:* a case row exists in DB with `userId`, `rolePack='defender'`, `title`, `createdAt`.
- **FR-011 — List cases.** Cases home shows all cases for the current user, sorted most-recently-updated first. *Acceptance:* list paginates 25 per page, scrolls at 60 fps.
- **FR-012 — View case.** Tapping a case opens the Briefcase view (4-pane dashboard). *Acceptance:* dashboard renders within 300 ms of tap with cached data, then live-updates if a run is active.
- **FR-013 — Rename case.** User can edit case title from the case header. *Acceptance:* update reflected in DB and list.
- **FR-014 — Delete case.** User can delete a case via swipe-to-delete on the home list. Soft delete — sets `deletedAt`. *Acceptance:* case disappears from list; recoverable for 30 days.

#### Ingest
- **FR-020 — Drive folder ingest.** User connects Google account, picks a Drive folder. Server walks the folder via Workspace MCP, fetches each file, hashes (SHA-256), OCRs if image/PDF, stores in `case_files` with `sourceType='drive'`. *Acceptance:* a folder of 10 files ingests in ≤ 30 seconds end-to-end.
- **FR-021 — Document scan ingest.** User opens device camera, scans multi-page document. App auto-crops, produces a single PDF per scan session, OCRs on-device, uploads to server with `sourceType='scan'`. *Acceptance:* a 4-page scan completes ingest in ≤ 15 seconds.
- **FR-022 — Direct file upload.** User picks a file from device storage. App uploads with `sourceType='upload'`. *Acceptance:* PDF, JPG, PNG, TXT supported up to 25 MB per file.
- **FR-023 — Idempotent ingest.** Duplicate files (same SHA-256) are not re-ingested or re-OCR'd. *Acceptance:* uploading the same file twice does not create two `case_files` rows.
- **FR-024 — Language detection.** Each ingested file gets `detectedLanguage` set by JurisdictionDetector. *Acceptance:* coverage of all CLDR top-50 languages.

#### Agent Run
- **FR-030 — Start a run.** User can start a run on a case by pressing the "Prepare" button on the case dashboard. Server creates a `runs` row, returns `runId`. *Acceptance:* `POST /v1/cases/:id/run` returns ≤ 200 ms.
- **FR-031 — Stream events.** Client opens an SSE connection to `/v1/runs/:runId/events`. Server emits events of types `run_started`, `planner_step`, `subagent_started`, `tool_call`, `tool_result`, `partial_result`, `subagent_completed`, `final_result`, `error`, `done`. *Acceptance:* first event arrives ≤ 2 seconds after run start; events arrive monotonically.
- **FR-032 — JurisdictionDetector first.** The first subagent in every run is JurisdictionDetector. Its result populates `cases.jurisdictionContext` and is passed to all subsequent subagents in their system prompt. *Acceptance:* `subagent_started` for JurisdictionDetector arrives before any other `subagent_started`.
- **FR-033 — Parallel subagent execution.** TimelineBuilder, EvidenceGapAuditor, CrossExaminationGenerator, PrecedentFinder run in parallel after JurisdictionDetector completes. *Acceptance:* all four `subagent_started` events fire within 500 ms of each other.
- **FR-034 — Cancel a run.** User can cancel a running run via the agent inspector. Server aborts in-flight model and tool calls. *Acceptance:* `done` event with `cancelled: true` arrives ≤ 2 seconds after cancel request.
- **FR-035 — Replay a run.** User can re-open a completed case and replay the agent stream from `run_events` storage. *Acceptance:* replay renders identically to live, at 4× speed.
- **FR-036 — Concurrent runs guarded.** A case may have only one active run. Attempting to start a second returns 409. *Acceptance:* contract test verifies.

#### Result Surfacing
- **FR-040 — Timeline view.** Renders `Timeline` artifact: events with date, title, source citation. Tapping an event opens the source PDF at the page anchor. *Acceptance:* renders ≤ 200 ms after `subagent_completed`.
- **FR-041 — Evidence gaps view.** Renders `Gaps` artifact: contradictions list with claim, source A, source B, severity. *Acceptance:* tap on each side shows source PDF.
- **FR-042 — Cross-exam questions view.** Renders `CrossExamSet`: question list with rationale, source, expected impeachment. *Acceptance:* user can mark a question as "added to prep" — persisted to `artifacts.data`.
- **FR-043 — Precedents view.** Renders `Precedents`: cases list with citation, jurisdiction, holding, why-relevant, source URL, **verifiedQuote**. Only verified precedents are shown. *Acceptance:* unverified precedents are dropped silently before render.
- **FR-044 — Source viewer.** Tapping any citation opens the source PDF with the cited span highlighted. *Acceptance:* highlight visible without manual scrolling on the cited page.

#### Settings & Compliance
- **FR-050 — Jurisdiction banner.** Settings shows the auto-detected jurisdiction for the current case (or aggregated across user's cases) and dynamically generates a one-paragraph compliance note. *Acceptance:* jurisdiction is never displayed as "unknown" — the JurisdictionDetector provides a confidence and a fallback.
- **FR-051 — Sub-processors disclosure.** Settings shows the list of sub-processors with versions: Anthropic, Google (Gemini + Workspace MCP), OpenAI (embeddings + GPT-5-mini), E2B, Tavily, Replit (hosting + Postgres). *Acceptance:* link to data-handling page from any sheet that uploads or sends.
- **FR-052 — Connect / disconnect Drive.** Settings exposes Google account connect/disconnect. Disconnect revokes the refresh token server-side. *Acceptance:* after disconnect, Drive picker is disabled and shows reconnect prompt.

#### Differentiating / Wow-Factor Features (Defender)

- **FR-060 — Contradiction Engine.** A dedicated subagent (`ContradictionEngine`) cross-references claims across police report, body-cam transcript timestamps, dispatch CAD logs, and witness statements. Each contradiction is rendered as a side-by-side card with both sources and a severity score. *Acceptance:* given a fixture set with a planted timestamp mismatch, the contradiction is surfaced within the run and exposes a tap-through to both source spans.
- **FR-061 — Rights Auditor (4th / 5th / 6th & equivalents).** A jurisdiction-aware subagent (`RightsAuditor`) reads ingested arrest paperwork and flags potential constitutional or statutory rights breaches (illegal stop / search, missing Miranda or local-equivalent caution, defective warrant, denied counsel). Each flag includes a binding precedent for the detected jurisdiction. *Acceptance:* tap a flag → opens precedent with verified quote; honors §10 jurisdiction context (US 4th Am, UK PACE Code C, India Article 22 / BNSS, EU CFR Art 6, etc.).
- **FR-062 — Brady / Disclosure-Gap Detector.** Scans the prosecution disclosure index against an expected-evidence checklist generated for the charge type and flags items that *should* exist but were not produced (body-cam for interaction X, lab report for substance Y, dispatch audio for time Z). *Acceptance:* missing-item card shows the inferred existence basis and the FRE / PACE / equivalent rule citation requiring its disclosure.
- **FR-063 — Mock Jury Simulator.** Spins up 12 AI juror personas (configurable demographic mix, defaults seeded for jurisdictional venue) who silently deliberate the user's defense theory and return per-juror reactions plus a final verdict distribution. *Acceptance:* runs end-to-end on a defense-theory string + case context in ≤ 60 s; output renders as 12 portrait cards with reactions and a deliberation transcript replayable via FR-035.
- **FR-064 — Live Courtroom Mode.** Defender places phone on counsel table; mic captures hearing audio, on-device VAD segments speakers, server transcribes near-real-time, and an `ObjectionWatcher` agent emits objection suggestions (hearsay, leading, foundation, relevance — jurisdiction-mapped) as silent haptic + lock-screen notification. *Acceptance:* objection suggestion arrives within 3 s of the qualifying statement; mic indicator always visible; on-device VAD prevents background uploads when no one is speaking.
- **FR-065 — Plea Outcome Simulator.** Given charge, jurisdiction, and (optional) defendant priors, runs a Monte Carlo over historical sentencing distributions (sourced via Tavily-bias to public sentencing databases / commission reports) to produce a plea-vs-trial outcome forecast with confidence band. *Acceptance:* output renders as a histogram + plain-English client-facing summary; every bucket cites its source dataset; never displayed as a guaranteed prediction (always probabilistic).
- **FR-070 — Glass Box Agent Theater.** Every run renders a live "agent tree" view: planner at root, subagents as children, each tool call as a leaf, with token-by-token streaming inside each leaf. Skia-rendered edges animate as data flows. Tap any node to inspect inputs, outputs, latency, and cost. *Acceptance:* a single screen renders the full DAG of a run within a 16 ms frame budget under load; replay mode reproduces the tree identically.
- **FR-071 — Time-Travel Agent Debugger.** From the agent tree, the user can rewind any run to any event index, fork a branch with edited inputs (different planner instructions, different model, different jurisdiction), and replay forward. Branches are persisted as `run_branches` rows linked to the parent run. *Acceptance:* a forked branch executes and streams its own events; the original run remains unchanged; both branches are diffable side by side.
- **FR-072 — Verifiable Citations Everywhere.** Every claim emitted by any subagent carries a `Citation = {sourceType, sourceId, span, verifiedQuote, verifiedAt}`. Tapping a citation in any view opens the underlying source (PDF page, video timestamp, web URL) with the cited span highlighted. Unverified citations are dropped before render. *Acceptance:* a contract test asserts every artifact field of type `claim` carries at least one verified citation; UI never renders an unverified claim.
- **FR-073 — AI vs AI Courtroom Simulator.** A `ProsecutionSimulator` subagent argues the same case as hypothetical opposing counsel, producing direct-exam outline, anticipated arguments, and likely rebuttals. The defender's brief is then "tested" against this simulated prosecution and a delta report highlights weak points in the defense theory. *Acceptance:* output renders as a paired transcript (defense move ↔ prosecution counter) and a ranked weakness list with cited authorities.

### 3.4 Non-Functional Requirements

#### Performance
- **NFR-P-001 — Cold start.** App opens to home in ≤ 1.5 seconds on iPhone 13 / Pixel 7.
- **NFR-P-002 — TTFB on SSE stream.** First agent event ≤ 2 seconds from `POST /run`.
- **NFR-P-003 — End-to-end run latency.** P50 ≤ 90 seconds for a 10-page PDF, 4 subagents, 1 verified-citation pass each.
- **NFR-P-004 — Frame rate.** UI maintains ≥ 60 fps under load on iPhone 13 / Pixel 7. Skia-rendered halos run on UI thread via Reanimated worklets.

#### Reliability
- **NFR-R-001 — Stream resumption.** If SSE connection drops mid-run, client auto-reconnects and resumes from last-seen event index.
- **NFR-R-002 — Idempotent run start.** Retrying `POST /v1/cases/:id/run` with same idempotency key returns the existing `runId`.
- **NFR-R-003 — Graceful agent failure.** A failing subagent emits `error` event; other subagents continue. Final dashboard shows the failed pane in an error state with a "Retry" affordance.
- **NFR-R-004 — Backend uptime target.** ≥ 99.5% during demo / beta.

#### Security
- **NFR-S-001 — Transport.** All client-server traffic over HTTPS; SSE over HTTPS only.
- **NFR-S-002 — Secret storage.** OAuth refresh tokens encrypted at rest using application-level AES-256-GCM keyed from `SESSION_SECRET`-derived KEK.
- **NFR-S-003 — Sandbox isolation.** Every E2B execution runs in a fresh Firecracker microVM. No filesystem persistence between runs unless explicitly opted in.
- **NFR-S-004 — PII handling.** Document text is sent to AI providers per their data-handling terms (Anthropic zero-retention available; Gemini per Google ToS; OpenAI zero-retention via Replit integration). Disclosed in §11.
- **NFR-S-005 — Local storage.** No raw OAuth secrets stored on device — only `expo-secure-store` for session token.

#### Usability
- **NFR-U-001 — Tap-to-result.** From cold start, the user can reach a streaming run in ≤ 4 taps.
- **NFR-U-002 — Reduced motion.** Honors `prefers-reduced-motion` automatically (Reanimated 4 default behavior).
- **NFR-U-003 — Large text.** UI scales with system font size up to 200%.
- **NFR-U-004 — Color contrast.** All text-on-background combinations meet WCAG AA (4.5:1).
- **NFR-U-005 — Haptic feedback.** Every meaningful event (run start, subagent complete, citation verified) triggers a discrete haptic.

#### Usability — Beginner & Onboarding
- **NFR-U-008 — First-launch boot sequence.** App boots with a particle-assemble logo and persona introduction (the personas literally introduce themselves). Skippable; runs once per install (`appState.firstRunCompleted`).
- **NFR-U-009 — Practice case.** The empty home offers a one-tap "Try a sample case" using local fixture data; runs entirely against canned events; never hits backend.
- **NFR-U-010 — Spotlight tour.** First run renders pulsing rings around the next action with a soft voice-over. Skippable; repeatable from Settings → "Replay tour".
- **NFR-U-011 — Plain-English mode.** Header toggle replaces specialist terms (e.g. "suppression motion", "discovery", "Brady material") with plain-language equivalents in every label, tooltip, and artifact summary. Glossary lives in `lib/i18n/glossary.{lang}.json`.
- **NFR-U-012 — Contextual helper.** A floating "What can I do here?" pill on every screen reveals 3 contextual actions in plain language.
- **NFR-U-013 — Plain-language confidence.** Numeric confidences render as labels (`Pretty sure` ≥ 0.85, `Likely` ≥ 0.6, `Worth checking` < 0.6) with color tokens; numeric value still available on long-press.
- **NFR-U-014 — Universal Undo.** Every destructive action emits a 5-second `<UndoToast>`; a session-scoped Recent Actions screen in Settings allows long-undo within session.

#### Usability — Modern Visual Language
- **NFR-U-015 — Liquid Glass everywhere.** Every floating surface uses native iOS 26 / Android 16 glass material; never a plain solid sheet.
- **NFR-U-016 — Aurora background.** Every primary screen renders a Skia mesh-gradient aurora that reacts to agent state (calm idle / rippling working / pulse on completion) via `AmbientReactor` (§5.7).
- **NFR-U-017 — OLED true-black + dynamic accent.** Background `#000000` on OLED devices; primary accent oscillates ±8% lightness with agent activity.
- **NFR-U-018 — Variable typography.** Variable-axis fonts (Inter Variable + Geist Variable) animate weight on emphasis, focus, and active states.
- **NFR-U-019 — 60 fps under load.** All Skia futurism (aurora, halos, neural tree, particles, holographic shimmer) runs on UI thread via Reanimated worklets and stays at 60 fps on iPhone 13 / Pixel 7. Auto-throttles to 30 fps when battery < 20%.
- **NFR-U-020 — Holographic AI marker.** AI-generated content carries a subtle iridescent gradient border (Skia shader, 6 s rotation) so authorship is *felt*, not just labeled.
- **NFR-U-021 — Sound design (opt-in).** 12-clip micro-sound bank (≤ 80 ms each) for taps, agent spawn, agent done, citation stamp, success, errors. Off by default; user opt-in in Settings.

#### Engine — Performance / Cost / Trust
- **NFR-E-001 — Hybrid model routing.** Every model call is routed through `ModelRouter` (§5.7); the chosen model and rationale are recorded with cost on `model_routing_decisions`.
- **NFR-E-002 — Speculative execution.** Subagents may speculatively start before the planner finalizes its plan; outputs from discarded branches are persisted to `speculative_branches` but not rendered.
- **NFR-E-003 — Semantic cache hit ratio.** ≥ 30% of LLM calls on a typical replayed run hit the semantic cache; cache key is `{role, promptEmbedding, toolArgsHash}`; persisted in `semantic_cache`.
- **NFR-E-004 — Streaming tool calls.** Every tool emits intermediate progress events (≥ 1 every 2 seconds for any operation > 4 seconds) so the Glass Box stays alive.
- **NFR-E-005 — Self-healing retries.** Subagent failures auto-retry up to 2× with smaller scope or alternate model; failures past the retry budget surface as `error` events with the reformulation history attached.
- **NFR-E-006 — Live cost transparency.** The Glass Box header always displays running cost-so-far in USD with model breakdown; final cost persisted on `agent_costs.totalUsd`.
- **NFR-E-007 — Constitutional gate.** Every artifact passes a values check before render; outputs violating role-pack constraints (e.g. unsupported jurisdiction guidance, unverified citations) are dropped and recorded in `policy_drops`.
- **NFR-E-008 — Audit bundle.** Every completed run produces a signed audit bundle (events + artifacts + citations + prompt versions + costs + model decisions) downloadable via `/v1/runs/:runId/audit`.
- **NFR-E-009 — OpenTelemetry tracing.** Every agent call, tool call, and model call gets a span; traces queryable for the Glass Box and exportable to any OTel-compatible backend.
- **NFR-E-010 — Quality judge.** A small evaluator LLM scores each subagent output against the role-pack rubric; outputs below threshold are auto-retried once before render; scores persisted in `quality_judgments`.
- **NFR-E-011 — Cost guardrails.** Per-user / per-tenant cost ceilings; runs nearing the ceiling automatically degrade to cheaper models; exceeding the ceiling halts the run with a clear in-UI message.
- **NFR-E-012 — Prompt versioning.** Every prompt is versioned and the version recorded on each run so outputs are reproducible; A/B variants supported via `prompt_versions`.
- **NFR-E-013 — Hierarchical memory.** Three tiers: working (per-run scratchpad), case (per-case via pgvector), tenant (per-user / per-agency). Retrieval is automatic by relevance; tenant scope is RLS-strict.
- **NFR-E-014 — Inter-agent collaboration.** Subagents may post messages to each other through `agentMessageBus`; messages persisted to `agent_messages` and visible in the Glass Box.
- **NFR-E-015 — Multi-strategy citation verification.** Citations pass through `verifierBank` which selects substring / semantic / temporal-anchor / image-cropbox strategy by source type; failures drop the citation silently.
- **NFR-E-016 — Replay regression.** Every demo-quality run is auto-saved to `replay_cases`; CI replays the suite and asserts identical or improved outputs against goldens.

#### Maintainability
- **NFR-M-001 — Type safety.** Zero `any` in client agent rendering code. All agent events Zod-validated at the boundary.
- **NFR-M-002 — Codegen-only DTOs.** No hand-written request/response shapes.
- **NFR-M-003 — Logger discipline.** Server uses `req.log` in handlers and singleton `logger` elsewhere; no `console.log`.

#### Portability
- **NFR-O-001 — Same APK/IPA path.** Both iOS and Android consume the same OpenAPI contract and the same agent protocol.
- **NFR-O-002 — Multi-language ready.** All user-facing strings in `i18n` resource bundle (English at launch).

#### Compliance
- **NFR-C-001 — Disclosure visibility.** A "Data & AI" link is reachable from every screen that initiates an AI call.
- **NFR-C-002 — Citation provenance.** Every citation displayed includes a tap-through URL to the verified source.

### 4.2 Use Cases

#### UC-01 — Prepare a case from Drive
**Actor:** Defender. **Trigger:** User taps "+ New Case" → "Pick from Drive."
**Preconditions:** User has connected Google account; Drive folder exists.
**Main flow:**
1. User selects a Drive folder.
2. App calls `POST /v1/cases` (returns caseId).
3. App calls `POST /v1/cases/:id/files/from-folder` with `driveFolderId`.
4. Server walks folder via Workspace MCP, fetches each file, OCRs if needed, persists.
5. Server returns ingestion progress events (per-file).
6. Once ingestion completes, app calls `POST /v1/cases/:id/run` with `rolePack: 'defender'`.
7. App opens SSE stream `/v1/runs/:runId/events`.
8. JurisdictionDetector runs first; jurisdiction is set on the case.
9. Four defender subagents run in parallel; events stream into 4-pane dashboard.
10. User taps a result; source viewer opens.
**Postconditions:** Case has artifacts; run history persisted; user can re-open and replay.

#### UC-02 — Prepare a case from camera scan
**Actor:** Defender. **Trigger:** User taps "+ New Case" → "Scan documents."
**Main flow:** As UC-01, but ingest step uses native scanner + on-device OCR; files uploaded with `sourceType='scan'`.

#### UC-03 — Prepare a case from a typed goal only
**Actor:** Defender. **Trigger:** User taps "+ New Case" → "Type a goal." Useful for follow-up runs on already-ingested cases.
**Main flow:** Server runs the planner with no files (or with previously-ingested files for the case). Planner may surface "No documents — please ingest first" if there is nothing to work on.

#### UC-04 — View a contradiction and tap through to source
**Actor:** Defender. **Trigger:** User taps an item in the Evidence Gaps pane.
**Main flow:** App opens the source viewer at the cited page anchor with the contradicting span highlighted in both source PDFs (split view).

#### UC-05 — Cancel a run mid-stream
**Actor:** Defender. **Trigger:** User taps "Cancel" in the agent inspector.
**Main flow:** Client calls `POST /v1/runs/:runId/cancel`. Server aborts in-flight tool calls and emits `done` with `cancelled: true`.

#### UC-06 — Replay a completed case
**Actor:** Defender. **Trigger:** User opens a previously-prepared case.
**Main flow:** App requests `GET /v1/runs/:runId/events?replay=true`; server streams stored events at 4× cadence; UI animates as if live.

#### UC-07 — See compliance posture for a foreign jurisdiction
**Actor:** Defender (working a case in a non-home country). **Trigger:** User opens Settings on a case whose jurisdiction differs from device locale.
**Main flow:** Settings shows the detected jurisdiction and a generated paragraph of guidance ("Aux États-Unis, public defenders should…").

### 5.1 Architectural Overview

Three-tier:

```
   ┌──────────────────────────────────────────────────┐
   │                  Briefcase (mobile)              │
   │  Expo SDK 54 · New Arch · Reanimated 4 · Skia    │
   │  NativeWind v4 · gluestack-ui v3 · FlashList v2  │
   │  AI SDK 5 · AI Elements · react-native-sse       │
   └──────────────────────┬───────────────────────────┘
                          │ HTTPS + SSE
                          ▼
   ┌──────────────────────────────────────────────────┐
   │           JusticeOS Engine (Hono on Node 20)     │
   │  Mastra orchestrator · Vercel AI SDK 5           │
   │  OpenAPI contract via @workspace/api-spec        │
   │  Drizzle ORM · @ai-sdk/* providers               │
   └──┬────────────┬────────────┬─────────────┬───────┘
      ▼            ▼            ▼             ▼
   Postgres   Workspace      E2B           Tavily
   + pgvector    MCP        Sandbox        REST
```

Key architectural decisions, locked:

| # | Decision | Rationale |
|---|---|---|
| 1 | One backend, two clients | Both Briefcase and Beat consume the same OpenAPI; only the role pack differs. |
| 2 | OpenAPI-first contract | All client/server types are codegen'd via `@workspace/api-spec`. Eliminates drift bugs. |
| 3 | SSE not WebSocket | One-way streaming is sufficient; SSE auto-reconnects, traverses every proxy, and integrates cleanly with Hono and AI SDK 5. |
| 4 | Mastra for orchestration | TypeScript-native; same framework as Replit Agent 3; pairs cleanly with AI SDK 5 via `streamVNext`. |
| 5 | MCP for Drive | Use Google's managed Workspace MCP server, not a bespoke REST wrapper. |
| 6 | E2B for sandboxed compute | Firecracker microVMs, ~150 ms cold start, install any Python pkg on demand. |
| 7 | pgvector in Replit Postgres | One database for both relational and vector data; one fewer system to operate. |
| 8 | Citation verifier mandatory | Every citation runs through an E2B verifier before display. Failed citations dropped silently. |
| 9 | JurisdictionDetector first | Universal architecture; no user-side country picker. |
| 10 | Demo auth = `x-demo-user` header | Real auth post-buildathon. |

### 5.5 Component Interface Descriptions

#### 5.5.1 `useAgentRun(runId)` — mobile

```ts
type AgentRunState = {
  runId: string;
  status: 'idle' | 'running' | 'completed' | 'cancelled' | 'error';
  jurisdiction?: JurisdictionContext;
  planner: { text: string; subagentsPlanned: string[] };
  panes: Record<string, AgentPaneState>; // keyed by subagent name
  events: AgentEvent[];                  // ordered by index
  cancel: () => Promise<void>;
};

useAgentRun(runId: string): AgentRunState;
```

Internally subscribes to SSE `/v1/runs/:runId/events`, validates each event with Zod, and updates a Zustand store. Components subscribe to a slice they care about (e.g. one pane subscribes to one subagent's events).

#### 5.5.2 `<AgentPane subagent="TimelineBuilder" />` — mobile

Props: `{ subagent: string; runId: string }`. Renders a Linear-violet rounded card with:
- header (subagent name, status pulse)
- live reasoning stream
- tool-call chips
- partial-result preview
- final result summary on completion

#### 5.5.3 Backend orchestrator — `streamRun(runId, rolePack)`

```ts
function streamRun(runId: string, rolePack: 'defender' | 'detective'): AsyncIterable<AgentEvent>
```

Yields events as JurisdictionDetector → Planner → parallel subagents → final_result → done. Persists every event to `run_events` for replay. Wires AI SDK 5 streaming to our event protocol via `streamWriter.ts`.

### 5.6 Error Handling Design

| Failure | Behavior |
|---|---|
| Mobile SSE drops | Auto-reconnect with `?since=<lastIndex>`. Replay stored events; resume tail. |
| Subagent throws | Emit `error` event; mark pane error; other subagents continue. |
| All subagents fail | Emit `done` with `status: 'error'`; show retry CTA. |
| Citation verifier fails | Drop the citation silently. Do not warn. |
| Drive 403 (token expired) | Force re-auth flow; redirect to Settings → Connect Drive. |
| E2B sandbox timeout | Mark single tool call failed; subagent may retry up to 2× then proceed without that tool result. |
| Postgres unavailable | API returns 503; mobile shows offline banner; queues run intent for retry. |
| LLM rate limit (429) | Exponential backoff; emit `tool_call` with `status: 'pending'` so the user sees what is happening; resume. |
| Network offline at startup | App opens to last cached cases (read-only); New Case is disabled with a banner. |

### 5.7 Cross-Cutting Design

#### Logging (server)
- In Hono route handlers: `req.log.info({...}, 'message')`.
- In non-request code: import the singleton `logger` from `lib/logger.ts`.
- Log levels: `error`, `warn`, `info`, `debug`.
- Never log: raw OAuth tokens, raw document text, raw model API keys.

#### Observability (server)
- Per-run trace id on every event and DB row (`runs.id`).
- `run_events` table is a queryable replay log.

#### Time and randomness
- All timestamps UTC ISO-8601 in storage and on the wire.
- Display in user device locale on mobile.
- Use cryptographic randomness for `runId`, `caseId`, `fileId` (`crypto.randomUUID()`).

#### Motion / Sound / Haptic / Ambient Systems (mobile)
- **MotionSystem (`lib/motion`)** — central spring & easing presets. 6 named tokens: `whisper`, `soft`, `bouncy`, `elastic`, `snap`, `dramatic`. All animations resolve through these tokens (no one-off curves). Reduced-motion fallback collapses every spring to a 120 ms opacity crossfade.
- **SoundSystem (`lib/sound`)** — 12-clip micro-sound bank (`tap`, `confirm`, `error`, `agentSpawn`, `agentDone`, `citationStamp`, `swipe`, `success`, `objection`, `verdictDrop`, `recordOn`, `recordOff`). All clips ≤ 80 ms; off by default; user opt-in in Settings (NFR-U-021).
- **HapticSystem (`lib/haptic`)** — 8-pattern taxonomy mapped to event types: `lightClick`, `success`, `warning`, `agentStart`, `agentDone`, `objectionDetected`, `citationVerified`, `verdictArrived`. Wraps `expo-haptics` with deterministic semantics.
- **AmbientReactor (`lib/ambient`)** — single subscription to agent state that drives the global Aurora background, tab-bar magnetic indicator, and ambient color-shift. One source of truth so animations never desync.
- **GlassBoxBus (`lib/glassbox`)** — typed event bus that mirrors the SSE stream into the Glass Box neural-tree renderer (A3) without a second subscription.

### 8.4 Schemas (Zod, in `lib/api-spec`)

```ts
Case = {
  id: string; userId: string; title: string; description?: string;
  rolePack: 'defender'; jurisdictionContext?: JurisdictionContext;
  language?: string; status: CaseStatus; createdAt: string; updatedAt: string;
}

CaseFile = {
  id: string; caseId: string; sourceType: 'upload'|'drive'|'scan'|'audio';
  driveFileId?: string; name: string; mime: string; sizeBytes: number;
  sha256: string; detectedLanguage?: string; createdAt: string;
}

Run = { id: string; caseId: string; rolePack: 'defender'; goal: string;
        status: 'pending'|'running'|'completed'|'cancelled'|'error';
        startedAt?: string; completedAt?: string; errorMessage?: string; }

JurisdictionContext = { country: string; region?: string;
                        legalSystem: 'common'|'civil'|'religious'|'mixed'|'customary';
                        language: string; confidence: number; }

AgentEvent =
  | { idx: number; type: 'run_started'; runId: string; rolePack: 'defender'|'detective'; goal: string }
  | { idx: number; type: 'planner_step'; text: string; subagentsPlanned: string[] }
  | { idx: number; type: 'subagent_started'; subagent: string; pane: number }
  | { idx: number; type: 'tool_call'; subagent: string; tool: string; args: object; status: 'pending'|'running'|'success'|'error' }
  | { idx: number; type: 'tool_result'; subagent: string; tool: string; resultPreview: string }
  | { idx: number; type: 'partial_result'; subagent: string; data: object }
  | { idx: number; type: 'subagent_completed'; subagent: string; data: object }
  | { idx: number; type: 'final_result'; data: object }
  | { idx: number; type: 'error'; subagent?: string; message: string }
  | { idx: number; type: 'done'; runId: string; cancelled?: boolean };
```

### 8.5 SSE Wire Format

Each event is one SSE `data:` line containing JSON of the `AgentEvent`. Server includes `id: <eventIndex>` for resumability (browsers and `react-native-sse` honor `Last-Event-ID`).

### 8.6 Errors

Standard JSON envelope:

```json
{ "error": { "code": "validation_error", "message": "title is required", "details": [...] } }
```

Codes: `validation_error`, `unauthorized`, `forbidden`, `not_found`, `conflict`, `rate_limited`, `dependency_unavailable`, `internal_error`.

### 8.7 Idempotency

`POST /v1/cases/:id/run` accepts `idempotencyKey` (UUID). Server stores it on `runs.idempotencyKey UNIQUE`. Replays return the existing `runId`.

### 8.8 Code Generation

Spec at `lib/api-spec/openapi.yaml`. Codegen via `pnpm --filter @workspace/api-spec run codegen`. Outputs:
- `lib/api-spec/src/generated/zod/` — Zod schemas
- `lib/api-spec/src/generated/react-query/` — React Query hooks (with `useSuspenseQuery` variants)
- `lib/api-spec/src/generated/types/` — TypeScript types

Both Briefcase (mobile) and the api-server (backend, for input validation) import from `@workspace/api-spec`.

## 9. Agent Architecture & Catalog (Defender Role Pack)

### 9.1 Orchestration Model

- **Mastra** orchestrator owns the lifecycle of a run.
- A **role pack** is `{plannerSystemPrompt, subagents[]}`. Briefcase loads the **defender role pack**.
- Mastra's `streamVNext` API is bridged to our SSE protocol via `streamWriter.ts`. The bridge maps Mastra events → `AgentEvent` types and assigns monotonic `idx`.
- The orchestrator persists every event to `run_events` synchronously before pushing to SSE — this enables exact replay (FR-035).

### 9.2 Planner Agent

| Field | Value |
|---|---|
| Model | Claude Sonnet 4.6 (Anthropic via Replit integration) |
| Temperature | 0.2 |
| Max output tokens | 1500 |
| System prompt template | `prompts/defender_planner.md` |
| Inputs | `{jurisdictionContext, goal, fileSummaries[]}` |
| Output | `{subagentsPlanned: string[], notes: string}` |

System prompt summary: "You are the planner for a criminal-defense preparation system. Given the user's goal and the documents available, decide which of the following subagents to run in parallel: TimelineBuilder, EvidenceGapAuditor, CrossExaminationGenerator, PrecedentFinder. ## 11. Security, Privacy & Compliance Design

### 11.1 Threat Model (STRIDE summary)

| Threat | Mitigation |
|---|---|
| **Spoofing** of user identity | Demo: header-based with seeded user. Production: Google sign-in + session cookie (HttpOnly, SameSite=Lax, Secure). |
| **Tampering** with documents in transit | TLS 1.3 only. Server validates SHA-256 on upload. |
| **Repudiation** of run actions | Every event persisted in `run_events` with monotonic `idx`. |
| **Information disclosure** to other users | Row-level filtering by `userId` on every query. Postgres role with no superuser. |
| **Denial of service** | Per-user run rate limit (post-launch). E2B sandboxed compute is bounded per call. |
| **Elevation of privilege** | Sandbox isolation: agents cannot reach the host filesystem; tool calls are explicit and validated. |

### 11.2 Data Classification

| Class | Examples | Handling |
|---|---|---|
| **Highly sensitive** | OCR'd document text (may contain PII), Drive refresh token | Encrypted at rest (refresh token AES-256-GCM); transmitted only over TLS; sub-processors disclosed |
| **Sensitive** | Run events containing model output | TLS only; retained 90 days |
| **Internal** | Run id, file id, sha256 | TLS only |
| **Public** | App version, model versions | n/a |

### 11.3 Encryption

| At rest | In transit |
|---|---|
| `users.googleRefreshToken` AES-256-GCM with KEK derived from `SESSION_SECRET` via HKDF | TLS 1.3 client ↔ server |
| Postgres at rest encryption (Replit-provided) | TLS to Anthropic / Gemini / OpenAI / E2B / Tavily |

### 11.4 Sub-processor Disclosure (in-app)

| Sub-processor | Purpose | Data shared |
|---|---|---|
| Anthropic (via Replit AI integration) | Planner + most subagents | Document text, system prompt, jurisdiction context |
| Google (Gemini, Workspace MCP) | Vision / OCR fallback / Drive read | Document images, file bytes |
| OpenAI (via Replit AI integration) | Embeddings, GPT-5-mini routing | Chunked text |
| E2B | Sandboxed Python execution | Document text, citation URLs |
| Tavily | Web search | Search query strings |
| Replit | Hosting + Postgres | All persisted data |

### 11.5 Compliance Postures by Region

| Region | Posture |
|---|---|
| US | ABA Formal Op 512 (lawyer-verified AI), Model Rule 1.6. Disclosed sub-processors. |
| EU / UK | GDPR / UK GDPR. Settings shows data-handling page with retention and sub-processors. |
| India | DPDP 2023 notice. |
| Brazil | LGPD notice. |
| All others | Generic "data sent to AI providers; review your bar / data-protection rules" notice. |

### 11.6 Audit & Forensics

- Every run has a UUID; every event has `idx + createdAt`; replayable.
- Hard delete schedule: soft-delete 30 days then hard-delete; refresh tokens revoked + deleted on disconnect.
- Logs scrubbed of raw tokens and raw document text.

| **Subagent** | A specialized agent invoked by the planner, with a defined toolset and structured output. |
| **Pane** | One quadrant of the 4-pane Briefcase view, bound to one subagent. |

| **E2B** | Sandboxed execution environment (Firecracker microVM). |
| **JurisdictionContext** | `{country, region?, legalSystem, language, confidence}` produced by the JurisdictionDetector. |
| **Verifier** | The E2B tool that fetches a cited URL and confirms the cited quote exists in the document. |
| **OCR** | Optical character recognition. |
| **SSE** | Server-Sent Events. |
